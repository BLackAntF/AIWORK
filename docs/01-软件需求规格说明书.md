## 1. 引言

### 1.1 编写目的

本文档旨在对智慧烹饪系统（智能加热炉 APP）的软件需求进行详细、明确的描述，为后续的概要设计、详细设计、编码实现及测试工作提供依据。本文档的预期读者包括：项目管理人员、软件开发人员、测试人员以及最终用户代表。

### 1.2 项目背景

随着物联网技术的快速发展和智能家居理念的普及，传统电磁加热设备在控制方式、安全监控、设备管理等方面已无法满足现代智慧厨房的需求。当前市场上的加热炉设备普遍存在以下痛点：

- **控制方式单一**：只能通过物理按键进行操作，无法实现远程控制
- **设备管理困难**：多台设备同时使用时，无法统一管理和批量操作
- **安全隐患突出**：设备异常时无法主动预警，存在过热、干烧等安全风险
- **功能扩展性差**：固定的火力档位和工作模式，难以适应多样化的烹饪需求

智慧烹饪系统（智能加热炉 APP）基于蓝牙 BLE 通信技术，通过移动端应用实现对 12KW 电磁机芯加热设备的智能控制，为用户提供设备组管理、精准火候调节、分区加热控制、异常报警、定时任务等核心功能。

### 1.3 术语定义

| 术语 | 定义 |
|------|------|
| BLE | Bluetooth Low Energy，蓝牙低功耗通信协议 |
| GATT | Generic Attribute Profile，通用属性配置文件 |
| UUID | Universally Unique Identifier，通用唯一识别码 |
| 设备组 | 由多个设备组成的逻辑分组，便于批量管理 |
| 火候档位 | 加热功率的等级划分，共 12 档（P1-P12） |
| 分区控制 | 对加热炉内区和外区进行独立或组合控制 |
| 分段控制 | 内区或外区内部的5段独立加热控制 |
| 机芯 | 电磁加热设备的核心控制单元 |
| 指令日志 | 记录所有操作指令和设备应答的日志系统 |

## 2. 总体描述

### 2.1 产品概述

智慧烹饪系统是一款基于 uni-app 框架开发的跨平台移动端应用，通过 BLE 蓝牙与智能加热炉设备通信，实现对电磁加热设备的远程智能控制。系统支持设备组管理、12 档火力调节、内外分区及分段独立控制、定时任务、异常报警、指令日志记录等功能，为用户提供安全、便捷、高效的智能烹饪体验。

### 2.2 产品功能

系统主要功能模块如下：

- **设备组管理**：创建、编辑、删除设备组，设备跨组拖拽，批量开关机
- **蓝牙设备管理**：扫描、连接/断开 BLE 设备，添加测试设备
- **设备操控面板**：开关机、12档火候调节（含快捷火候）、分区控制
- **定时任务**：倒计时定时（1-60分钟）、暂停/继续/取消、定时结束自动关机
- **异常报警**：6种异常类型上报、持续震动提醒、一键关机解除
- **指令日志**：完整记录所有收发指令和系统事件
- **测试模拟**：模拟报警测试，无需真实设备即可验证功能

### 2.3 用户特征

| 用户类型 | 特征描述 | 使用频率 | 技术水平 |
|---------|---------|---------|---------|
| 普通用户 | 家庭厨房使用者，通过APP控制加热炉 | 每天使用 | 低 |
| 厨师用户 | 餐饮行业专业厨师，多设备同时操作 | 高频使用 | 中 |
| 测试人员 | 开发测试人员，验证设备功能 | 项目周期内 | 高 |

### 2.4 运行环境

#### 2.4.1 硬件环境

- 支持蓝牙 4.0 及以上的 Android 设备（Android 5.0+）
- 支持蓝牙 4.0 及以上的 iOS 设备（iOS 9.0+）
- 12KW 电磁机芯智能加热炉（配备 BLE 通信模块）

#### 2.4.2 软件环境

- **前端框架**：uni-app（Vue 2 / Vue 3 双模式兼容）
- **开发工具**：HBuilderX 3.x+
- **通信协议**：12KW 电磁机芯串口明文协议（波特率 9600）
- **打包平台**：Android / iOS / 微信小程序

### 2.5 约束与假设

**约束条件：**

- 设备通信必须基于 BLE 蓝牙协议，无法使用 WiFi 或移动网络
- 单台手机同一时间只能与一台设备进行实时交互
- 设备指令采用 ASCII 明文字符串格式，以 `>` 开头
- APP 所有数据采用本地存储，不依赖云端服务

**假设条件：**

- 用户设备已开启蓝牙和定位权限
- 加热炉设备已通电且处于可发现状态
- 蓝牙信号覆盖范围内无严重电磁干扰

## 3. 功能需求

### 3.1 设备组管理模块

#### 3.1.1 创建设备组

- 功能描述：用户可以创建新的设备组，用于分类管理多个设备
- 输入：设备组名称（字符串，不能为空）
- 处理流程：
  1. 用户点击右上角 [+] 按钮 → 选择"创建设备组"
  2. 弹出输入框，用户输入组名称
  3. 校验名称非空后，将新组插入到设备组列表末尾之前（"未分组设备"始终在最后）
  4. 显示"创建成功"提示
- 输出：新的设备组被创建，初始为空组

#### 3.1.2 编辑组名

- 功能描述：用户可以修改设备组的名称
- 输入：新的组名称
- 约束："未分组设备"为系统默认组，不可重命名
- 输出：组名称被更新

#### 3.1.3 删除设备组

- 功能描述：用户可以删除指定的设备组
- 处理流程：
  1. 用户选择删除设备组
  2. 弹出确认对话框，提示组内设备将移回"未分组设备"
  3. 用户确认后，将组内所有设备移至"未分组设备"
  4. 删除该设备组
- 约束：至少存在一个可删除组时才显示删除选项

#### 3.1.4 设备拖拽排序

- 功能描述：用户长按设备卡片后可以拖拽，实现设备在组内或跨组移动
- 触发方式：长按设备卡片 1 秒后进入拖拽模式
- 视觉反馈：
  - 被拖拽的设备原位置透明度降低
  - 跟随手指的浮动卡片显示设备信息
  - 目标组高亮显示
- 输出：设备位置更新

#### 3.1.5 批量控制

- 功能描述：对设备组内所有在线设备执行批量开机或批量关机
- 触发条件：设备组展开状态且组内有设备
- 处理流程：
  1. 用户点击"全部开机"/"全部关机"按钮
  2. 弹出确认对话框
  3. 用户确认后，遍历组内所有在线设备执行开关机操作
  4. 显示执行结果：已操作设备数量
- 输出：组内在线设备状态同步更新

### 3.2 蓝牙设备管理模块

#### 3.2.1 蓝牙扫描

- 功能描述：扫描周围的 BLE 蓝牙设备
- 前置条件：手机蓝牙已开启
- 处理流程：
  1. 初始化蓝牙适配器
  2. 开始蓝牙设备扫描
  3. 实时发现设备并去重添加到列表
  4. 显示设备名称、MAC 地址、信号强度（RSSI）
  5. 扫描持续 5 秒后自动停止
- 输出：设备列表（名称、MAC、信号强度）
- 异常处理：
  - 蓝牙未开启：提示"请开启手机蓝牙"
  - 未发现设备：显示"未发现蓝牙设备"并提供重新搜索

#### 3.2.2 添加设备

- 功能描述：将扫描到的蓝牙设备添加到设备列表中
- 输入：用户选择的蓝牙设备
- 处理流程：
  1. 用户从扫描结果中选择设备
  2. 创建设备对象，初始化默认状态（P0、待机、在线）
  3. 将设备添加到"未分组设备"组
  4. 显示"已添加 {设备名}"提示
- 默认属性：
  - 设备名称：蓝牙设备名称
  - 图标：🔥
  - 火候档位：P0
  - 状态：待机
  - 离线状态：否

#### 3.2.3 添加测试设备

- 功能描述：添加模拟设备用于功能测试，无需真实硬件
- 设备类型：4 种类型循环（测试灶具-A、测试灶具-B、测试烤炉-A、测试烤炉-B）
- 命名规则：{类型}-{序号}，如"测试灶具-A-1"
- 设备标识：MAC 地址格式 `TEST:00:00:00:XX`
- 特殊处理：测试设备指令发送失败不弹出错误提示

#### 3.2.4 删除设备

- 功能描述：从设备列表中移除设备
- 操作方式：点击设备卡片右上角的关闭按钮
- 处理规则：
  - 在"未分组设备"中删除：直接移除
  - 在其他组中删除：设备移回"未分组设备"组

### 3.3 设备操控模块

#### 3.3.1 开关机控制

- 功能描述：控制设备的启动和停止
- 开机指令：`>AFO`
- 关机指令：`>AFF`
- 设备应答：`>OK`
- 约束条件：
  - 设备离线时禁止操作，提示"设备离线"
  - 已开机时禁止重复开机
  - 已关机时禁止重复关机
- 输出：设备状态更新（运行中 / 已停火）

#### 3.3.2 12档火候调节

- 功能描述：调节设备的加热功率，共 12 个档位
- 档位范围：P1（最小）~ P12（最大/爆炒）
- 档位指令：
  - P1~P9：`>AF1` ~ `>AF9`
  - P10：`>AFA`
  - P11：`>AFB`
  - P12：`>AFC`
- 调节方式：
  - 加减按钮：逐档调节（支持长按连续调节，间隔 200ms）
  - 快捷火候：一键切换到常用档位（小火P3 / 中火P6 / 大火P9 / 爆炒P12）
- 视觉反馈：
  - 12段火候条，不同区间不同颜色（P1-P4 蓝色 / P5-P8 橙色 / P9-P12 红色）
  - 档位数字实时更新

#### 3.3.3 分区整体控制

- 功能描述：对加热炉的内区和外区进行整体控制
- 三种模式：
  - 仅内区工作：指令 `>ACI`
  - 仅外区工作：指令 `>ACX`
  - 内外全区工作：指令 `>ACL`
- 默认模式：内外全区同时工作

#### 3.3.4 分段独立控制

- 功能描述：对内区或外区的5个分段进行独立控制
- 内区分段指令：`>AA1` ~ `>AA5`
- 外区分段指令：`>AB1` ~ `>AB5`
- 用途：实现不同位置的差异化加热

### 3.4 定时任务模块

#### 3.4.1 添加定时

- 功能描述：设置倒计时任务，时间到后自动关机
- 时长范围：1 ~ 60 分钟（整分钟选择）
- 输入：用户选择的分钟数
- 处理流程：
  1. 用户点击"+ 添加定时任务"
  2. 弹出选择列表（1~60 分钟）
  3. 用户选择后启动倒计时
  4. 显示"定时 XX 分钟"提示

#### 3.4.2 定时控制

- 暂停/继续：用户可以暂停或继续倒计时
- 取消定时：用户可随时取消定时任务
- 倒计时显示：分:秒格式（MM:SS），不足两位补零

#### 3.4.3 定时结束处理

- 功能描述：定时时间到达后的自动处理
- 处理流程：
  1. 倒计时归零
  2. 自动发送关机指令 `>AFF`
  3. 火力档位归零
  4. 显示"时间到！已自动关机"提示
  5. 触发长震动提醒

### 3.5 异常报警模块

#### 3.5.1 异常类型

系统支持 6 种设备异常类型：

| 异常码 | 指令 | 描述 | 严重级别 |
|--------|------|------|---------|
| AE1 | `>AE1` | 机芯超温 | 高 |
| AE2 | `>AE2` | 线盘超温 | 高 |
| AE3 | `>AE3` | 过流保护 | 高 |
| AE4 | `>AE4` | 电流异常 | 中 |
| AE5 | `>AE5` | 锅具异常 | 中 |
| AE6 | `>AE6` | 线盘测温开路 | 中 |

#### 3.5.2 异常上报处理

- 触发方式：设备主动推送异常码
- 处理流程：
  1. 接收到 `>AE1` ~ `>AE6` 异常码
  2. 在报警区域新增报警卡片（异常名称 + 描述 + 时间）
  3. 启动持续震动提醒（每 500ms 震动一次）
  4. 新增一条指令日志
  5. 全局事件通知，首页自动跳转到该设备控制页
- 报警列表：按时间倒序排列，最新在前

#### 3.5.3 报警解除

- 解除方式：用户点击报警卡片上的"关机"按钮
- 处理流程：
  1. 发送关机指令 `>AFF`
  2. 火力档位归零
  3. 移除该条报警
  4. 记录日志
  5. 若无待处理报警，停止震动

#### 3.5.4 模拟报警

- 功能描述：测试预警功能，模拟设备异常上报
- 入口：首页底部"测试预警"区域
- 操作流程：
  1. 选择模拟的目标设备
  2. 选择异常类型（6 种）
  3. 自动跳转到控制页并显示报警
- 用途：功能演示和测试验证

### 3.6 查询功能模块

#### 3.6.1 温度查询

- 功能描述：查询设备三路温度数据
- 指令：`>AT?`
- 返回格式：`>A XX XX XX XX XX XX`（12 位十六进制，每 4 位一路温度）
- 解析方式：每 4 位十六进制对应一路温度值
- 显示：在日志中记录三路温度值

#### 3.6.2 电参数查询

- 功能描述：查询设备电压、功率、电流
- 指令：`>AP?`
- 返回格式：`>A XX XX XX XX XX XX`（12 位十六进制）
- 解析：前 4 位=电压，中间 4 位=功率，后 4 位=电流
- 显示：在日志中记录电参数数据

### 3.7 指令日志模块

#### 3.7.1 日志记录

- 记录内容：
  - 所有发送的控制指令（含指令码和描述）
  - 所有接收的设备应答和上报
  - 系统事件（进入页面、连接状态变化等）
- 记录信息：时间戳 + 消息内容
- 排序方式：按时间倒序排列，最新在顶部

#### 3.7.2 日志类型

| 类型 | 前缀 | 示例 |
|------|------|------|
| 发送指令 | 操作描述 + [指令码] | "快捷火候：小火 [>AF3]" |
| 执行成功 | ✅ | "✅ 指令执行成功" |
| 温度数据 | 🌡️ | "🌡️ 温度: 1路=xxx 2路=xxx" |
| 电参数 | ⚡ | "⚡ 电压=220V 功率=12000W" |
| 异常报警 | ⚠️ | "⚠️ 机芯超温 [>AE1]" |
| 系统事件 | 文字描述 | "进入控制页面" |

## 4. 非功能需求

### 4.1 性能需求

- **响应速度**：指令发送后界面反馈时间 < 100ms
- **连接速度**：蓝牙设备连接建立时间 < 3s
- **扫描效率**：5 秒内完成设备扫描
- **页面切换**：页面跳转响应时间 < 500ms
- **动画流畅度**：UI 动画帧率 ≥ 30fps

### 4.2 可靠性需求

- **指令可靠性**：关键指令（开关机、报警关机）确保发送成功
- **数据一致性**：设备状态与实际状态保持一致
- **异常恢复**：蓝牙断开后自动重连提示
- **数据持久化**：设备列表和分组信息本地存储，重启不丢失

### 4.3 易用性需求

- **操作直观**：主要功能（开关机、调档）在首屏可直接操作
- **学习成本**：新用户 5 分钟内可掌握基本操作
- **反馈明确**：每个操作都有视觉或震动反馈
- **错误提示**：操作失败时有清晰的错误说明

### 4.4 兼容性需求

- **平台兼容**：支持 Android 5.0+、iOS 9.0+
- **屏幕适配**：适配不同尺寸的手机屏幕
- **蓝牙版本**：兼容 BLE 4.0 及以上
- **框架兼容**：Vue 2 / Vue 3 双模式

### 4.5 安全性需求

- **权限申请**：仅申请必要的蓝牙和定位权限
- **数据安全**：所有数据本地存储，不上传云端
- **安全保护**：超温、过流等异常自动告警和保护
- **操作确认**：批量操作、删除操作需二次确认

### 4.6 可维护性需求

- **代码规范**：遵循 uni-app 和 Vue 开发规范
- **模块化**：蓝牙通信独立封装（ble.js）
- **注释完整**：关键功能和算法有清晰注释
- **可扩展性**：预留后续功能扩展接口


### 5.2 主要用例描述

#### 用例 1：设备扫描与添加

| 项目 | 内容 |
|------|------|
| 用例名称 | 扫描并添加设备 |
| 参与者 | 用户 |
| 前置条件 | 手机蓝牙已开启 |
| 主事件流 | 1. 用户点击"添加设备"<br>2. 系统开始扫描蓝牙设备<br>3. 显示扫描结果列表<br>4. 用户选择目标设备<br>5. 设备添加到"未分组设备" |
| 后置条件 | 设备出现在设备列表中 |
| 异常流 | 蓝牙未开启 → 提示开启蓝牙<br>未发现设备 → 显示空状态并提供重新搜索 |

#### 用例 2：设备分组与批量控制

| 项目 | 内容 |
|------|------|
| 用例名称 | 设备组管理与批量控制 |
| 参与者 | 用户 |
| 前置条件 | 至少有一台设备已添加 |
| 主事件流 | 1. 用户创建设备组<br>2. 长按设备拖拽到目标组<br>3. 展开设备组<br>4. 点击"全部开机"按钮<br>5. 确认后组内设备全部开机 |
| 后置条件 | 设备按组组织，组内设备状态同步 |
| 异常流 | 组名重复 → 提示重名<br>空组删除 → 确认后直接删除 |

#### 用例 3：火候调节

| 项目 | 内容 |
|------|------|
| 用例名称 | 12档火候调节 |
| 参与者 | 用户 |
| 前置条件 | 设备已开机且在线 |
| 主事件流 | 1. 用户进入设备控制页<br>2. 点击 + / - 按钮调节档位<br>3. 或点击快捷火候一键切换<br>4. 系统发送对应指令<br>5. 火候条和档位数字实时更新 |
| 后置条件 | 设备加热功率变化 |
| 异常流 | 设备离线 → 禁用操作并提示<br>已达最高档 → + 按钮无响应 |

#### 用例 4：异常报警处理

| 项目 | 内容 |
|------|------|
| 用例名称 | 设备异常报警与处理 |
| 参与者 | 用户、设备 |
| 前置条件 | 设备处于连接状态 |
| 主事件流 | 1. 设备检测到异常并上报异常码<br>2. APP 接收并解析异常码<br>3. 显示报警卡片，持续震动提醒<br>4. 用户点击"关机"按钮<br>5. 发送关机指令，移除报警，停止震动 |
| 后置条件 | 设备关机，报警解除 |
| 异常流 | 多个异常同时发生 → 按列表分别显示<br>报警期间用户未操作 → 持续震动 |

#### 用例 5：定时任务

| 项目 | 内容 |
|------|------|
| 用例名称 | 定时关机 |
| 参与者 | 用户 |
| 前置条件 | 设备处于运行状态 |
| 主事件流 | 1. 用户设置定时时长<br>2. 倒计时开始<br>3. （可选）用户暂停/继续倒计时<br>4. 时间到达 → 自动发送关机指令<br>5. 震动提醒用户 |
| 后置条件 | 设备关机 |
| 异常流 | 用户取消定时 → 停止计时 |

## 6. 数据需求

### 6.1 设备数据结构

```
Device {
  name: String           // 设备名称
  icon: String           // 设备图标 emoji
  mac: String            // MAC 地址 / 设备唯一标识
  deviceType: String     // 设备类型（cooker / grill）
  isTest: Boolean        // 是否为测试设备
  power: Int             // 当前火力档位（0-12）
  statusText: String     // 状态描述文本
  offline: Boolean       // 离线状态
  zoneMode: String       // 分区模式（inner / outer / all）
  innerSegment: Int      // 内区分段（0-5，0为关闭）
  outerSegment: Int      // 外区分段（0-5，0为关闭）
}
```

### 6.2 设备组数据结构

```
DeviceGroup {
  name: String           // 组名称
  expanded: Boolean      // 是否展开
  devices: Device[]      // 组内设备列表
}
```

### 6.3 报警数据结构

```
Alarm {
  alarmCmd: String       // 异常指令码
  alarmLabel: String     // 异常名称
  title: String          // 报警标题
  desc: String           // 报警描述
  timestamp: Number      // 时间戳
}
```

### 6.4 日志数据结构

```
LogItem {
  time: String           // 时间（HH:MM:SS）
  msg: String            // 日志消息
}
```
